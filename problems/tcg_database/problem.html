Introduction:

This problem is centered around writing a program that manipulates a database of cards from a a Trading Card Game. The game is Magic the Gathering. Deep knowledge of the rules of the game is not required, but a limited understanding of the main ressource of the game, of what is known as "Cube", and of the anatomy of the card is useful. Let's start with the main ressource the game uses:


Mana:

"Mana" is the main ressource of the game. It would be the equivalent of money in Monopoly, and while you would use your hard-earned cash to buy buildings in Monopoly, you use up to 6 different types of "Mana" to use cards in this game. The six types correspond to 6 colors, and for each color there is an associated letter and icon:

  • - White: the letter associated is : W
  • - Blue: the letter associated is : U
  • - Red: the letter associated is : B
  • - Black: the letter associated is : R
  • - Green: the letter associated is : G
  • - Colorless: the letter associated is : C

You can think of these as different currencies, to continue the comparison with Monopoly. Some costs can be paid in any color, in that case a number is used: this means that the cost is 3 Mana of any color. For more information on Mana, click here.


Cube:

We will know briefly talk about what is known as Cube in the context of Magic, the Gathering. There are many different ways of playing with these cards. The closest comparison would be the different formats of Chess: Bullet, Classical, Chess960, Fog of War, ect...

Cube is one such format. Cards will be assembled into a list, of usually 360 cards, by what one could call a "Cube Designer". The selections is made from the pool of over 24000 different cards printed. After carefully choosing the cards wanted in the list, in a process that can take hours, or days, the design is complete.

To actually play with the Cube, there are many options, one of which is the following: Each player picks 90 cards randomly from the pool of cards. Those are the cards that that player will be able to play with for the rest of the game. Those cards are usually handed out as 6 packs of 15 cards.


Card Anatomy:

We will be manipulating a database that contains, for each card, all of its relevant information. We will quickly look at the anatomy of a card, to better manipulate the database afterwards.



Let's take a quick look at the elements on this card, by going over the different elements of it's anatomy:

  • - name:
    • The name of each card is a string. That string is unique to the card, and it is defining: if two cards have two different names, they are different. If two cards have the same name, they are equal. The only differences you may have between two cards with the same name are aesthetic.
  • - image:
    • The image is an illustration that conveys no rules information about the card. It is purely aesthetic.
  • - type_line:
    • The type_line conveys information that helps to categorize the card. Different types have different rules. The closest equivalent would be the suits in a normal game of cards. There is a large, but finite amount of types possible.
  • - oracle_text:
    • This block of text defines the rules that this card must follow. It is not a defining trait: two cards can have the same oracle_text, yet be different cards. The closest equivalent would be, for a game of chess, the definition of all rules that a given piece must follow in terms of movement. Not all cards have an oracle_text.
  • - power and toughness:
    • These two values are relevant to combat in the game, you could consider them an extension of the oracle_text. They are not necessarily numbers, and not all cards have a power and toughness. For the most part, only cards with "Creature" in their type_line have a power and toughness.
  • - set:
    • Sets are lists of cards that are printed together to form an organized collection. As far as is needed for this exercise, the set is aesthetic. They are the official "Cube Lists", built by the designers of Magic the Gathering.
  • - mana_cost:
    • Refers to the ressource cost to play the card. As we said earlier, each color has a corresponding letter. This allows the representation costs as strings. For instance, in the case above the cost would be: "{3}{R}".
  • - color_id:
    • This refers to the number of different colors costs that appear anywhere on the card. In this case, there is a {R} symbol in the mana_cost, and two {U/G} symbols ( which mean pay with Green or Blue, by the way) in the oracle_text. That means that the color identity of this card is Blue Red Green.
  • - cmc:
    • This is not directly displayed on a card but is also relevant information. CMC, or Converted Mana Cost, is the total amount of mana required to pay the price of the card, regardless of color. It is an integer. For instance, in the above case, the cost is {3}{R}, so three of any color and one red, and that means the CMC is 3+1 = 4.
    In an other example, if the cost is {3}{U/B}{B/G}{C}{U}{U}, the cost would be 3 + 1 U/B + 1 B/G + 1 C+ 2 U = 8 If you know the mana_cost, you can calculate the CMC, but if you have the CMC, you can't necessarily know the Mana Cost.


This covers the minimum required theory regarding the rules of the game.



Technical Information:


We will cover some technical details here. For instance, if you have problems that relate to the database file, or information on the cards, you make find some useful information here:


https://scryfall.com/docs/api/cards

It's technical documentation on the database we will now be working with. As written earlier, you will be working with a database of all cards. They are gathered in a .json file, and the link to that file is in the required files, above.

You have been provided with a script that deals with translating the json file into data that you can work with. The file "tcg_database_methods" contains methods you will be using in during the exercise. You do not need to write anything into that file. It is preferable that you write, in your own script, at the top : 


import tcg_database_methods as tcg

There are two functions that you can call in that script:
  • - tcg.jsonreader()
    • This function takes one parameter, the location of the .json file containing the card database. It returns a list of dictionaries. Each dictionary contains all the data relevant to a given card.
  • - tcg.listofurltowebsite()
    • This function takes one parameter, a list of strings corresponding to URLs of images. It returns nothing, and will write an .html page with all the images drawn in sequence left to right, top to bottom.


Problem:

Given that we have access to all cards ever printed, thanks to the .json file, we would like to generate a random Cube List of cards. Here are the specifications:

  • - On demand, we need to be able to generate a Cube of random cards, aka a random list of a given number of cards.
  • - On demand, we should be able to control how many cards go into the list.
  • - For a given Cube, we should be able to simulate picking 15 random cards from that list to form one of the 6 packs one would assign to a player.
  • - Leave the design space open for more specifications.


In time more specifications will be added, especially concerning which cards are assigned to the list of the Cube.
